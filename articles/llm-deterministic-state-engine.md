## LLMエージェントの「状態」はなぜ壊れるのか

個人プロジェクトでLLMエージェントに継続的なタスク管理を任せていると、次のような壊れ方に何度も遭遇します。

- コンテキストウィンドウが要約された結果、数日前に確定した決定事項が消えている
- エージェントが「タスクXは80%完了しました」と自己申告するが、それを裏付ける成果物がどこにもない
- 「その設定は先週すでに変更しました」と言うが、実際には一度も変更されていない（幻覚された状態）

架空の例を1つ挙げます。203X年6月某日、個人プロジェクトの管理をエージェントに任せていたとして、こんなやり取りが起きたとします(以下は説明用に創作した架空の会話・架空の数値です)。

> 人間: 「先週のタスクボードのP1項目、終わった?」
> エージェント: 「はい、ドラフトAの3項目はすべて完了しています」
> (実際には2項目しか完了しておらず、残り1項目は着手すらされていない)

これは能力不足でも悪意でもなく、**状態(state)をLLMの生成結果として持たせている限り原理的に避けられない**問題です。プロンプトの中、会話履歴の中、あるいはLLMの「記憶」の中に状態を置くと、状態は次の3つの経路で壊れます。

1. 圧縮・要約によって失われる
2. 「自己申告」という検証不可能なプロセスを経由する
3. そもそも「尤もらしい文字列」を生成する装置が、状態の真実性を担保する主体になってしまう

## 設計: 状態をコードに追い出す

個人プロジェクトとして開発しているAI OSでは、この問題に対して「LLMに状態を持たせない」という設計を採っています。構成は3層です。

1. **事実層(append-only JSONLイベントログ)**: 起きたことを1行ずつ記録する。修正・削除はせず、新しい事実を追記するだけ。
2. **投影層(純Pythonスクリプト `sync.py`)**: このイベント列から「今の状態」を機械的に計算する。**LLMは一切呼ばない**。標準ライブラリのみで実装されており外部依存はゼロ。
3. **表示層(タスクボード)**: 投影層が生成した決定論的なMarkdownを、次のセッションでLLMエージェントが読み込む。

架空の例で言うと、イベントログは次のような行の集まりです(架空データ)。

```jsonl
{"ts":"2031-04-02T09:00:00Z","type":"task","entity":"draft-report-x","state":"in-progress","ball":"me","priority":"P1"}
{"ts":"2031-04-03T10:15:00Z","type":"task","entity":"draft-report-x","state":"done","ball":"os","priority":"P1"}
```

2行目のタイムスタンプが1行目より新しい場合、「今の状態」は機械的に2行目に決まります。これが設計の核心で、**「最新 = ts 最大」というルールを人間にもLLMにも判断させない**という点です。判断するのはタイムスタンプの大小比較だけです。

## 決定論の実証: 実装を見る

「LLMが介在しない」という主張は、口だけでは検証できません。実装を見てみます。

まずタイムスタンプのパースです。ISO8601以外の形式やUnixエポックなどは弾き、比較可能なdatetimeオブジェクトに正規化します。

```python
# sync.py:23-29
def parse_ts(s):
    """ISO8601(…Z / +09:00 / naive)を tz-aware datetime に。失敗は None(=不正ts)。"""
    try:
        dt = datetime.fromisoformat(str(s).replace("Z", "+00:00"))
        return dt if dt.tzinfo else dt.replace(tzinfo=timezone.utc)
    except Exception:
        return None
```

次に「最新」を決める中核ロジックです。文字列比較ではなく時刻比較で最新を決め、同時刻のイベントは後勝ちで単純に上書きせず、リストとして保持しておきます。

```python
# sync.py:98-109
def project(rows):
    latest = {}
    for e in rows:
        key = (e.get("_module", ""), str(e.get("type", "")), str(e.get("entity", "")))
        ts = str(e.get("ts", ""))
        dt = parse_ts(ts) or datetime.min.replace(tzinfo=timezone.utc)
        cur = latest.get(key)
        if cur is None or dt > cur["dt"]:
            latest[key] = {"ts": ts, "dt": dt, "events": [e]}
        elif dt == cur["dt"]:
            cur["events"].append(e)
    return latest
```

同一tsのイベントを消さずに配列として溜めておくのが、後述するCONFLICT検知の伏線になっています。

そして「本当に決定論か」はテストで機械的に検証しています。回帰テストスクリプトの最初のチェックは、同じ入力に対してスクリプトを2回実行し、出力ファイルがbyte単位で一致するかを比較するというものです。

```bash
# selftest.sh:19-21
D="$(fresh d1)"; ( cd "$D"; "$PY" scripts/sync.py . >/dev/null 2>&1; cp state/now.md "$TMP/a";
  "$PY" scripts/sync.py . >/dev/null 2>&1; cp state/now.md "$TMP/b" )
cmp -s "$TMP/a" "$TMP/b" && ok "決定論 byte一致" || ng "決定論 byte不一致"
```

これが通らなくなったら「決定論」という主張自体が崩れたことを意味するので、変更のたびに機械的に再検証されます。「AIエージェントに勝手にスクリプトを直させたら、いつの間にか決定論が壊れていた」という事故を、レビューの注意力ではなく回帰テストで防いでいる形です。

## 矛盾は隠さず「CONFLICT」として出す

「最新 = ts最大」だけでは足りません。もし同じタイムスタンプで矛盾する2つの事実が記録されていたらどうするか。多くの実装はここで「配列の最後を勝者にする」「後から読んだ方を採用する」といった暗黙のルールでどちらかを勝たせてしまいます。これは一見便利ですが、**silent failure**(黙って片方を握りつぶす)の温床です。

架空の例で示します。

```jsonl
{"ts":"2031-05-10T00:00:00Z","type":"task","entity":"vendor-decision-y","state":"done","ball":"me","priority":"P1"}
{"ts":"2031-05-10T00:00:00Z","type":"task","entity":"vendor-decision-y","state":"backlog","ball":"ext","priority":"P3"}
```

同一entity・同一タイムスタンプで、片方は「完了」、もう片方は「未着手(backlog)」と言っています。投影層は `(state, ball, priority)` の組をタイムスタンプが同一のイベントすべてから集め、異なる組が複数存在する場合は「機械では決定不能」と判断してCONFLICTとして扱います。架空の出力例は次のようになります。

```markdown
## ⚠ CONFLICTS(決定論で解決不能・人手要)
- pj-example/task/vendor-decision-y: 同一ts 2031-05-10T00:00:00Z で 2通りの(state,ball,priority)
  [('backlog','ext','P3'),('done','me','P1')] → 人手解決要
```

重要なのは、これが**盤面のどこに切り詰められても消えない**設計になっている点です。タスクボードには文字数上限があり、超過分は切り詰められますが、CONFLICTセクションは常に先頭に描画され、切り詰め対象から明示的に除外されます。「矛盾を見なかったことにする」のが一番危険な失敗モードだと考えているので、目立つ場所に、目立つ形で残すようにしています。判断を下すのはあくまで人間で、機械は「判断できない」という事実だけを正確に報告します。

## 副産物: モデルが変わっても状態層は壊れない

この設計にはもう1つ利点があります。状態を計算するのがLLMではなく決定論的なPythonスクリプトである以上、**その日たまたま使っているLLMが何であっても、状態層の振る舞いは変わりません**。エージェントを別のモデルに乗り換えても、盤面の生成ロジックは1文字も変わらないので、モデル移行のたびに状態管理をやり直す必要がありません。

さらに、セッションが消えても平気です。LLMエージェントとの対話はセッション単位で消えていく前提で設計していますが、真実の記録はバージョン管理下のイベントログにしかないので、新しいセッションを開始した瞬間に同じスクリプトを1回走らせるだけで、同じ盤面が寸分違わず復元されます。「前回の会話を覚えているかどうか」に一切依存しない継続性です。
