ローカル音声入力ツール [Koe（声）](https://github.com/dkamehat/Koe) の続きです。
前回の記事の最後に「次はローカルで会議の同時通訳を検証中」と書きました。それが動いたので共有します。

## 何を作ったか

**Koe Interpreter** は、PCで鳴っている相手の声（Zoom / Meet / 動画）を
**全部ローカルで** 文字起こし → 翻訳し、さらに **「こう返すといい」という返答案まで出す**
CLIツールです。`interpreter.py` 一本で動きます。

- リポジトリ: https://github.com/dkamehat/Koe （MIT）
- Windows用。音声は一切マシンの外に出ません（テーゼ: **ローカル必須 / 誰でも安全**）。

![Koe Interpreter demo](https://github.com/dkamehat/Koe/raw/main/docs/interpreter-demo.gif)

*デモではテスト音源として YouTube（BBC Learning English）の英会話を流していますが、想定している
本来の使い方は Zoom / Meet などの**実際の通話・面談**です。「スピーカーに出ている音」を捕まえる
仕組みなので、相手の声が鳴っていれば同じように字幕化・翻訳・返答案が出ます。*

一言でいうと「英語の通話で、相手が何を言っているか字幕で分かり、
**自分が英語で何と返せばいいかの下書きが出る**」ツールです。音声出力（読み上げ）はしません。
ゴールはあくまで **理解 + 返答サジェスト** です。

## なぜ作ったか

英語の通話やMTGで、聞き取りと「どう返すか」を同時にやるのがしんどい。
かといって、会話の中身をクラウドの文字起こしサービスに流すのは仕事上できない。
Koe の土台（faster-whisper + ローカルLLM）がもうあるので、入力を **マイク → スピーカーの
ループバック** に差し替えれば、同じ「ローカルで完結」の制約のまま作れるはず——という発想です。

## 構成

3スレッドを分離して、GPU処理が音声キャプチャを止めないようにしています。

```
 ループバック捕捉 ─► 生ブロック ─► 区間判定(エネルギーVAD) ─► 発話
   (capture)                         (segment)              ─► 文字起こし ─► 字幕
                                                              (transcribe)    │
                                                                              ▼
                                                       ローカルOllamaで翻訳 + 返答案生成
```

- **入力**: WASAPI ループバックで「スピーカーに出ている音」をそのまま録る。
- **STT**: Koe 本体と同じ faster-whisper。区間（発話の切れ目）は短い無音 or 長さ上限で切る。
- **翻訳**: `--to ja` などでローカル Ollama に投げる（本体の③整形とは別系統で、こちらは
  言語ガードを外して「翻訳していい」モード）。
- **返答案（APEX）**: `--suggest`（F9）または `--auto-suggest`（疑問符を検知して自動）で、
  直近の会話＋`--role`（役割）＋`--context`（履歴書/JD/アジェンダ等のファイル）を踏まえた
  返答案を、相手の言語＋自分の言語のグロス付きで出す。全部ローカル。

## ハマったところ（知見）

### 1. `soundcard` がこのマシンで「無音」しか返さない

ループバック録音の定番ライブラリ `soundcard` を使ったら、**全部ゼロ（完全無音）** が
返ってきて長時間ハマりました。音は確かに鳴っているのに、です。原因は環境側——
Realtek + ASUS の AIノイズキャンセル仮想デバイスの組み合わせ。**`PyAudioWPatch`**
に替えたら同じ出力先で一発で録れました（rms 0.06）。
ループバックはライブラリと音声スタックの相性が出る、という教訓。

### 2. ローカルLLM翻訳に中国語が漏れる

日本語訳のはずなのに、qwen2.5:**7b** がときどき簡体字や未翻訳の英単語を混ぜてくる
（`专长`、`更多贡献` など）。プロンプトで「中国語禁止・全語を訳せ」と縛り、断片化を避ける
ために区間長を抑えても、稀に貫通する。最終的に二段構え：

- **モデルを qwen2.5:14b に上げる**と漏れが消えた（`何样的`→「どのような」等、サンプル全滅）。
  16GB GPU で文字起こし用 turbo と同居できる。1訳あたり ~2.7s（7b は ~1.6s）。
- それでも保険として、**ターゲット言語を踏まえた「簡体字専用文字」検知ガード**を実装
  （`--to ja` のとき簡体字だけの文字が出たら再生成。`--to zh` のときは当然スルー）。

### 3. 速度は「測ってから」——ストリーミングは要らなかった

「リアルタイム性」を上げたくて逐次ストリーミングASRを実装しかけたのですが、先に計測したら
**字幕の体感遅延は 0.7〜1.3秒**で、ボトルネックはそこではなかった。区間長の上限
（`--max-seg`、既定8秒）を調整するだけで十分実用的。Koe 本体でも「遅延の犯人は推論じゃなく
`localhost` 解決だった」という前科があり、**測ってから直す**を今回も徹底しました。

### 4. VADのしきい値を手で調整したくない（今回の新機能）

発話/無音の判定は音量（RMS）がしきい値を超えるか、という単純なVADなんですが、この
しきい値はマシンや音源で最適値が違い、`--threshold 0.01` のように手で合わせるのが面倒でした。
そこで **起動時にループバックのノイズフロアを ~1秒測って、しきい値を自動で決める** ように：

- ノイズフロアは **低パーセンタイル（35%点）** で取る。こうすると、校正中にうっかり
  喋りが混じっても（ループバックに「静かにして」とは言えない）、そのスパイクで
  しきい値が吊り上がりにくい。
- それを `×マージン` して、`[0.005, 0.03]` でクランプ。デジタル無音（RMS≒0）でも
  しきい値が0に落ちないし、校正中に大音量が鳴っても上限で抑える（上限は発話のRMS
  〜0.06より下に置き、うるさい瞬間に校正しても発話を弾かないようにしている）。

純粋ロジック（測定値→しきい値）は I/O から切り出してユニットテスト済み。
`--threshold` を明示すればそれが優先、`--no-calibrate` で従来の固定値に戻せます。

## 使い方

```powershell
# 相手の声を字幕化（既定スピーカーを自動キャプチャ、しきい値は自動校正）
python interpreter.py --to ja

# 返答案つき（F9で下書き、--role で役割、--context で資料を読ませる）
python interpreter.py --to ja --suggest --role "PM interview, be concise" --context brief.md

# 疑問符を検知して自動で返答案を並べる
python interpreter.py --to ja --auto-suggest

# きれいな翻訳には 14b 推奨
python interpreter.py --to ja --ollama-model qwen2.5:14b
```

Ollama が必要です（本体の整形で使うのと同じサーバー）。動いていなければ字幕は原文のみになります。
