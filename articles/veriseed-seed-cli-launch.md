## 「とりあえずDBに何か入れる」が地味にしんどい

新機能を作り始めるとき、マイグレーションは通した。でもDBが空っぽで、画面を開いても何も表示されない。一覧も詳細も真っ白。

よくある対処:

- **手書きのfixture/seedスクリプト** — 動くけど、スキーマが変わるたびに壊れる。「だいたい同じデータ」は「同じデータ」じゃないので、CIがたまに落ちる原因になる。
- **Faker** — ローカルで動くけど、スキーマを知らない。外部キーの挿入順は自分で考える必要があるし、親が存在しないFKを生成して普通に落ちる。
- **AIチャットに「サンプルデータちょうだい」と聞く** — 一回きりの小さいサンプルなら、これが一番早い。実際、自分もよくやる。ただし数十行を超えると省略されがちだし、聞くたびに違うデータが返ってくるし、CIで自動的に回すことはできない。

「動くデータがサクッと、しかも毎回同じ形で欲しいだけなのに」という気持ちから、**Veriseed** というCLIを作った。

## npx veriseed で何が起きるか

```sh
npx veriseed seed --schema schema.prisma
```

Prismaスキーマを読んで、外部キーが必ず解決する順番（親→子）でテーブルを埋め、そのまま流せるSQLを1本のトランザクションとして出力する。

```sql
BEGIN;
INSERT INTO "User" (...) VALUES (1, 'aoi.ito1@example.com', 'Aoi Suzuki', 'Fukuoka', 'Japan');
INSERT INTO "Post" (..., "authorId") VALUES (1, ..., 1); -- authorId は実在する User を参照
COMMIT;
```

アカウント登録もAPIキーも不要。インストールすればオフラインでも動く。

## 一番効くのは「毎回同じデータが出る」こと

`--seed`（デフォルト42）を指定すると、**今のスキーマから、何度実行してもバイト単位で同じデータ**が出る。

これがCIだと地味に効く。手書きfixtureはスキーマ変更のたびに壊れて放置されがちだけど、Veriseedは「今のスキーマから毎回作り直す」ので、fixtureが腐るという概念自体がなくなる。

大量データが欲しいときは `--rows` を増やすだけ。ページングやN+1問題はそれなりの件数がないと再現しないので、これも地味にありがたい。

## 「それ、Faker + トポロジカルソートじゃない？」

正直、コアの部分は概ねその通り。ただ、Fakerを使うと外部キーの解決順・親の主キーからのFK選択・行内の整合性を、スキーマが変わるたびに自分で書き直す羽目になる。Veriseedはそれをスキーマから機械的にやる、というだけの話。派手な魔法は入っていない。

## 今できること／これから

いま無料でOSS公開しているのは、この「FK整合・決定論的」なコア部分だけ。名前・都市・国が矛盾しない程度の軽い整合性はあるけど、生成AIによる"本当にリアルな値"は今のところ入っていない。

その先の「ローカルAIモデルでよりリアルな値を作る」「PIIマスキング」「本番データの一部だけを参照整合性を保ったまま抜き出す（subsetting）」といったPro機能は、**まだ作っていない**。需要があるか確かめるための先行予約を$19（正式ローンチ後は$38予定）で受け付け中で、形にならなければ全額返金する。
